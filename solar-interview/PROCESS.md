## A Decision You Rejected

When reading the original stats logic, something felt off — it was calling `->get()` before counting, and from prior SQL experience I knew pulling a full result set into PHP memory just to count it was wrong. The original approach loaded every matching project row into a PHP Collection, then filtered and counted in PHP.

I rejected that in favor of DB-level aggregates (`->count()`, cloned query builders). The database can count and filter far more efficiently than PHP can — and the PHP approach degrades badly as the table grows. This wasn't in the TODOs; I caught it while reading the code critically.

## What Surprised You

Two bugs not listed in the TODOs:

1. `$project->submitted_at->format('M d, Y')` — would crash on any draft project where `submitted_at` is null. Found during code review before running the app.
2. `pending_projects` was being calculated and passed from the controller but never rendered in the blade template — silently dropped data.

In hindsight, the better workflow would have been to run the app visually first and let it crash, then trace the errors back to their cause. Instead I audited the code statically first and caught them that way. Both approaches work, but running it first would have made the bugs more obvious and given me a tighter feedback loop.

## Code You Don't Fully Understand

Two areas I had to study before I could confidently explain:

**`clone $query`** — Eloquent query builders are mutable objects. Calling `->where()` on a builder modifies it in place, stacking conditions on the original. Without `clone`, every stat aggregate would inherit constraints from the previous one and return wrong counts. I studied this, verified I understood it, and can explain it.

**`julianday()`** — SQLite-specific function that converts a datetime to a floating-point day count. Subtracting two `julianday()` values gives the difference in days; multiplying by 24 converts to hours. I studied this, verified the math against known data, and can explain it.

## How I decomposed the work across agents

I split the work into three agents with hard file-level boundaries:

- **Agent A (Backend)** — owned `DashboardController.php` exclusively. Responsible for all query logic: date filtering, SQL aggregates, avg approval time, and pagination.
- **Agent B (Frontend)** — owned `dashboard.blade.php` exclusively. Responsible for the date filter form, stat cards, table rendering, empty state, and pagination nav.
- **Agent C (Data Integrity)** — read-only access to the SQLite database. Wrote findings to `DATA_INTEGRITY_REPORT.md` only. No application code touched.

The three agents are independent by design — A and B share a data contract (the variables passed to the view) but never touch the same file. C has no dependency on A or B at all. This means all three can run in parallel without stepping on each other.

---

## What guardrails I set up and why

**File-level ownership** — each agent was given an explicit list of files it could and could not touch. This prevents agents from "helpfully" refactoring code outside their scope and breaking adjacent work.

**Query scoping rule** — Agent A was instructed that all queries must start from `$ahj->projects()`, never `Project::all()` or `Project::query()`. This enforces multi-tenancy at the agent instruction level, not just in code review.

**No full table loads** — Agent A was explicitly told to use DB aggregates (`->count()`, `->avg()`) and never load a full result set into PHP memory for counting. This prevents a class of performance bugs that are easy to introduce and hard to spot in review.

**Date filter safety** — Agent A was told to default to no filter (show all data) on blank input, never crash. This prevents a 500 error becoming the default UX for a new user who hasn't set a date range.

**No inline JS** — Agent B was told no `<script>` blocks. Keeps the frontend pure Blade/HTML and prevents scope creep into JS territory that would require a separate verification pass.

**Agent C read-only** — Agent C had no write access to application code. Findings go only to `DATA_INTEGRITY_REPORT.md`. This prevents a data analysis agent from "fixing" data issues unilaterally without human review.

## What I'd do differently next time

**Explicit data contract between A and B up front.** I defined the view variables (`$stats`, `$projects`, `$ahj`) implicitly through the controller. Next time I'd write a one-page data contract before either agent starts — what keys are in `$stats`, what type `$projects` is (Paginator, not Collection), what format `avg_approval_time` is in. This eliminates any guesswork on the frontend side.

**Reviewer agent as a formal step, not an afterthought.** Agent C was scoped to data integrity only. A separate reviewer agent that reads Agent A and B output and critiques it against the spec would catch things like: does `withQueryString()` actually preserve the date params through pagination? Is the `avg_approval_time` format string human-readable for edge cases like 0 hours? I'd make this a required gate before visual verification.

**Seed a known dataset for verification.** Spot-checking against live seeded data means the expected values aren't known in advance — you're eyeballing plausibility. A small fixed dataset (e.g., 5 projects with known `submitted_at` and `approved_at` values) would let you calculate expected avg approval time by hand and compare exactly.

## What You'd Do Next

The dashboard is functional but has several follow-up items that didn't fit within scope:

**Avg approval time formatting.** The current output is raw hours (e.g. `127.5 hours`). A human-readable format like `5 days, 7 hours` would fit the stat card better and be less likely to line-break mid-label at normal screen widths. This is a one-line controller fix.

**Date range picker component.** The native `<input type="date">` works but the UX is poor — it renders differently across browsers and isn't intuitive for a date range. A lightweight Tailwind-compatible date picker would be the right follow-up ticket.

**UI polish.** Two small improvements noticed during visual review: zebra stripes on the project table (`even:bg-gray-50` in Tailwind) for readability, and a displayed dash (`—`) in the submitted date column when `submitted_at` is null rather than a blank cell.

**CRUD was deliberately out of scope.** This is a read-only reviewer dashboard by design. Adding write access would require a separate spec and a different permission model — not a follow-up polish ticket.
